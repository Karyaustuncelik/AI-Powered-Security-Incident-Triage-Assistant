# AI-Powered Security Incident Triage Assistant

AI-Powered Security Incident Triage Assistant is a local-first security analysis web application that helps analysts review suspicious activity, prioritize incidents, and generate human-readable explanations from structured findings.

This project is designed as a serious portfolio piece rather than a chatbot demo. It combines deterministic backend triage logic with LLM-generated explanations so that risk decisions are not left entirely to a language model.

## Why this project matters

Security teams do not only need raw logs. They need a workflow that can:

- group suspicious activity into analyst-friendly incident records
- identify why an event looks risky
- assign a severity and investigation priority
- explain the result in plain language

This project models that workflow with a hybrid architecture:

- rule-based scoring for consistency and explainability
- structured incident objects for clean API design
- optional LLM output for narrative summaries
- fallback explanations so the UI still works if the model fails

## Core features

- Live incident dashboard with summary cards and queue management
- Rule-based triage engine with score, severity, and priority mapping
- Detected indicators for each incident
- Incident detail panel with score breakdown and technical facts
- Persistent analyst review workflow backed by SQLite
- Filtering by severity, priority, event type, actor, entity, search text, and time range
- LLM explanation endpoint with template fallback
- Local synthetic dataset with realistic cybersecurity scenarios
- Automated backend test suite

## Tech stack

- Frontend: React, Vite, TypeScript
- Backend: FastAPI, Python, Pydantic
- Data layer: local curated JSON dataset
- Workflow state: SQLite-backed analyst review store
- AI layer: mock provider by default, OpenAI-compatible and Azure OpenAI-ready provider interface
- Testing: Python `unittest`

## What the app does

The application simulates a security operations triage workflow for suspicious identity, access, privilege, and API abuse events.

Supported incident families in the current version:

- multiple failed login attempts
- repeated authentication failures
- risky sign-in pattern
- abnormal access outside normal hours
- impossible travel pattern
- unusual privilege escalation
- suspicious API activity

For each incident, the system produces:

- category
- severity
- priority
- detected indicators
- technical facts
- suggested action
- optional LLM explanation

## Triage logic

The backend does not ask the LLM to decide risk from scratch.

Instead, it follows a deterministic scoring flow:

1. Start with an event-type-specific base score
2. Add weighted signal contributions
3. Map the final score to a severity
4. Convert severity and escalation indicators into a priority

Current severity mapping:

- `0-29` -> `low`
- `30-54` -> `medium`
- `55-74` -> `high`
- `75+` -> `critical`

Example indicators:

- `repeated_failed_logins`
- `geo_anomaly`
- `privilege_abuse`
- `unusual_time_access`
- `api_spike`
- `admin_after_hours_activity`
- `permission_change_surge`
- `risky_signin_context`

## LLM explanation flow

The explanation layer receives structured incident data, not raw unbounded text.

The payload includes:

- event type
- category
- severity
- priority
- detected indicators
- technical facts
- score breakdown
- suggested action

The provider is expected to return JSON with:

- `short_explanation`
- `why_risky`
- `recommended_action`

If the LLM provider fails or is not configured, the backend falls back to a template explanation so the product remains usable.

## Dataset strategy

The current version uses a curated synthetic dataset stored in:

- `backend/data/incidents.json`

Analyst workflow state is stored separately in:

- `backend/data/triage_assistant.db`

The dataset is generated by:

- `backend/scripts/generate_synthetic_incidents.py`

This keeps the project easy to run locally while still modeling realistic scenarios such as impossible travel, privilege escalation, API spikes, and after-hours admin activity.

The next evolution path is to import public security logs, normalize them into canonical incident objects, and keep the same triage pipeline on top.

## Local setup

### 1. Clone or open the project locally

Use VS Code or your preferred editor in the project root.

### 2. Backend setup

```bash
cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### 3. Frontend setup

```bash
cd frontend
npm install
```

### 4. Environment variables

Copy `.env.example` into `.env` if you want custom local settings.

Default local frontend API base URL:

```env
VITE_API_BASE_URL=http://127.0.0.1:8000
```

Optional LLM settings:

```env
LLM_PROVIDER=mock
LLM_API_URL=
LLM_API_KEY=
LLM_MODEL=
LLM_TIMEOUT_SECONDS=20
```

Provider modes:

- `mock`
- `openai_compatible`
- `azure_openai`

## Running the project

### Start the backend

```bash
cd backend
source .venv/bin/activate
uvicorn app.main:app --reload
```

### Start the frontend

```bash
cd frontend
npm run dev
```

Then open the local Vite URL shown in the terminal, usually:

- [http://127.0.0.1:5173](http://127.0.0.1:5173)

## Regenerating the dataset

```bash
cd backend
source .venv/bin/activate
python scripts/generate_synthetic_incidents.py
```

## Running tests

```bash
cd backend
source .venv/bin/activate
python -m unittest discover -s tests -v
```

## Azure roadmap

This project is intentionally local-first for the MVP. A strong next step would be:

- Frontend on Azure Static Web Apps
- Backend on Azure Container Apps
- LLM explanation provider on Azure OpenAI
- Monitoring with Azure Monitor / Application Insights
- Persistent incident storage in Azure Database for PostgreSQL

That path keeps the current architecture mostly intact while making the system production-shaped.
